## 1. 本次会话做了什么（按阶段）

### 阶段一：三段式流水线（拆 → 改 → 写）
把原来只会「拆大纲」的单阶段 runner，升级成**同一套 `run.mjs` 按「活动阶段」驱动**的三段式：
- ① 拆大纲 `step1_break_outline`：`章节(.txt)` → `拆大纲(.md)`。**task 模式**：无需语义上下文，可多对话、乱序、可并发（concurrency=3）；每5章合1请求；每100章换对话。
- ② 改编 `step2_adapt`：`拆大纲(.md)` → `02_改编(.md)`。**novel 模式**：整本同一会话；每5章1请求。
- ③ 生成正文 `step3_generate`：`02_改编(.md)` → `03_正文(.md)`。**novel 模式**：整本同一会话；**逐章1请求**。
- 切阶段：`node run.mjs --stage <id>` 或改 `config.json` 的 `activeStage`。
- 每本一组子目录，文件名跨阶段一致（`第NNN章_xxx`）。

### 阶段二：下载器接入 + 迁移 + 编排器 + 出错即停
- **下载器**＝番茄 `TomatoNovelDownloader`（Rust，zhongbai2333）。`--server` 暴露 HTTP API（`127.0.0.1:18423`）：`POST /api/jobs{book_id}` 下载、`GET /api/jobs` 看 `progress.saved_chapters`、`GET /api/search?q=`。**无榜单接口**。
- `scripts/download.mjs`：确保 server 在跑 → 逐本下载（txt+散装，落 `data/.staging`）→ **ingest** 规整成 `<本>/章节/第NNN章_标题.txt`。
- **迁移**：D: 旧素材库16本 COPY 进 `data/00_raw_chapters`，`改编大纲`→`拆大纲`（`scripts/migrate-library.mjs`，幂等）。
- **编排器** `scripts/pipeline.mjs`：严格顺序 下载→拆→改→写；每阶段后**阶段门**用 `run.mjs --dry-run` 复算 `__PENDING__`/`__FAILED__`，皆0才放行。
- **出错即停 `failStop`**：novel 模式某章最终失败→停掉该本后续章（章级停本）；阶段有失败→不进下一阶段（阶段停全）。

### 阶段三：合并单一项目 + 自动更新 + 自动取榜单 + 上下文保证
- **合并**：旧 runner、D: 素材/工具、novel_pipeline 三者合一。下载器 exe 移进项目 `scripts/tomato/bin/`；说明书/写作指南→`docs/`，龙门上传助手→`tools/`。删旧功能（8787 web 控制台 server.mjs/web/、prompt-queue、launch-web、run-forever 守护）。主入口＝`pipeline.mjs`。
- **下载器自动更新**（`scripts/update-downloader.mjs`）：比对 GitHub Releases 最新 tag，有新版就**直接下载新 exe 原子替换**（不走会弹 TUI 卡死的 `--self-update`）。节流6h；server 占用时跳过。开机由计划任务 `TomatoSelfUpdate`（`scripts/register-tasks.ps1` 注册）、打开项目/每次跑 pipeline 也检测。**实测把 v2.4.11 自动升级到 v2.4.12 成功**。
- **自动取新人榜**：`download.mjs` 默认抓 `https://fanqienovel.com/rank/0_1_24`（服务端渲染，正则 `\/page\/(\d+)` 提 book_id），合并 `scripts/tomato/books.txt`（手动补充）去重后下载。`--no-rank` 关、`--rank <url>` 改。

## 3. 逻辑研究：怎么「保证上下文」（核心讨论）
**拆大纲**：每章独立、无需上下文 → 可多对话、乱序、并发，最快（concurrency=3）。
**改编 / 生成正文**：必须按正确顺序、且需要上下文。风险：① ChatGPT 会话被 KeepChatGPT 自动刷新或崩溃会丢；② 超长本超出模型上下文窗口、早期章节被挤出记忆。

采用的方案＝**两者结合**（`run.mjs` 的 `buildContext`）：
1. **同会话**：每本开唯一会话，整本顺着发，自然保留对话记忆。
2. **提示词自带「前情」锚点**：发某章前，从磁盘读它**前 K 章的产物**（K=`contextPrevOutputs`，默认1；`contextMaxChars` 截尾部=最近剧情）拼进提示词。→ 被刷新/超长本/崩溃都不丢眼前上下文。
3. **顺序强制**：`failStop` 保证 N 章必须先有 N-1 产物才发；锚点天然依赖前一章。
4. **断点从磁盘重建**：续跑时锚点直接读盘，无「接缝」。
> 代价：每请求多耗 token（用户接受，要最稳）。可调 `contextPrevOutputs`/`contextMaxChars`。

## 4. 架构与文件
```
novel_pipeline/
  开始.cmd                双击=检测下载器更新 + 看各阶段进度 + 常用命令提示
  README.md  docs/（说明书.txt 写作变现指南.txt 本文件）  tools/（龙门小说上传助手.exe）
  data/00_raw_chapters/<本>/{章节,拆大纲,02_改编,03_正文}   data/.staging/（下载暂存）
  legacy/                 用户把旧 D:\快穿小说项目准备 整体并到了这里（备份）
  logs/
  scripts/
    pipeline.mjs          编排器：下载→拆→改→写，阶段门，出错即停
    download.mjs          驱动下载器 API + 自动取新人榜 + ingest
    update-downloader.mjs 下载器自动更新（GitHub 直下替换）
    register-tasks.ps1    注册开机计划任务 TomatoSelfUpdate
    run-pipeline.ps1      确保 Chrome(9222) + 跑一次 pipeline
    status.mjs            各阶段进度速览（dry-run 复算）
    migrate-library.mjs   D:→data/ 迁移（含 --verify 校验门）
    tomato/  bin/TomatoNovelDownloader.exe  config.yml(txt+散装+save_path)  books.txt
    gpt-outline-runner/   run.mjs  lib/{files,select,chatgpt}.mjs  config.json  launch-chrome.ps1  HANDOFF.md
```
- `run.mjs` 核心：`applyStage` 把活动阶段覆盖进顶层 cfg；`contextScope=task` 走工作池、`=novel` 走 `novelWorker`（整本一会话+前情锚点+章级 failStop）。
- `lib/files.mjs`：`listChapters`(按 inputExt)、`buildPlan`(applySelection/失败统计)、claim/write/skip。`lib/select.mjs`：弧识别(`arcNameOf`)+前N章。
- 配置全在 `scripts/gpt-outline-runner/config.json`（stages 字典 + 顶层默认 + failStop）。

## 5. 怎么用（每月）
1. （可选）把新人榜30本的 book_id/链接补进 `scripts/tomato/books.txt`（默认已自动取榜单）。
2. 确保调试 Chrome 9222 已登录三个自定义 GPT（拆/改/写）。
3. 跑：`node scripts/pipeline.mjs`（或 `powershell scripts\run-pipeline.ps1` 会先拉起 Chrome）。
   - 跳过下载：`--no-download`；从某阶段起：`--from step2_adapt`。
4. 某阶段没全完成会**停下**，排查后重跑该阶段或 `--from`。
- 看进度：`node scripts/status.mjs` 或双击 `开始.cmd`。
