# xxsbooks 提示词与项目说明

## 项目作用

`xxsbooks` 是一个本地批量队列工具，用 Chrome 自动化把本地小说章节按顺序发送给 ChatGPT GPTS，并把 GPTS 回复保存到本地。

当前项目用于中文快穿小说工作流：把多个来源小说拆成小世界素材，再改编、重排章节节奏，生成新的快穿小说正文。

## 当前核心流程

现在不是原来的“单章拆、单章改、单章写”，而是：

```text
原文 3 章
  -> 拆成 3 章原剧情细纲
  -> 改编并重排成 2 章新剧情细纲
  -> 本地拆成 2 个单章大纲文件
  -> 正文 GPTS 逐章生成
```

对应目录：

```text
input
  -> output/01_dagang
  -> output/02_adapt
  -> output/02_adapt_chapters
  -> output/03_zhengwen
```

这样做的目的：

- 打破原文“第 1 章对第 1 章”的结构痕迹。
- 提高每章信息密度，让正文节奏更快。
- 重新设计章末钩子和爽点位置。
- 减少正文照着旧章节机械扩写的感觉。
- 仍然保持单次输入不太长，避免整小世界一次处理导致漏细节。

## 用户当前需求

用户要把多个来源小说拆成“小世界”基本单位，再重新打乱组合成新的快穿小说。由于来源小说的系统、奖励、能力、记忆、升级、世界观规则不统一，所以改编时必须统一成以下规则：

- 每个小世界独立成立。
- 每个小世界可以任意打乱、任意插入新的快穿小说。
- 女主统一叫“姜浅予”。
- 不继承上个世界的记忆、能力、奖励、道具、系统商城、积分、空间、感情线或主线伏笔。
- 原系统、奖励、好感度、升级等设定，全部本土化为本世界内部资源、证据、原身记忆、职业能力、信息差、人脉、资金或合理线索。
- 保留原剧情的核心矛盾、利益冲突、事件推进、压迫感、反转功能、爽点功能和悬念功能。
- 正文要自然、流畅、像人写的小说，避免系统播报、任务面板、世界线修正等科幻感。

## 提示词文件说明

```text
00A_每次输入前缀_拆原剧情细纲.md
```

用于拆大纲 GPTS 的每次输入前缀。它强调：一次输入 1-3 个连续原文章节，只做逐章原剧情拆解，不改编、不重排、不合并章节。

```text
00B_每次输入前缀_改编重排.md
```

用于改编 GPTS 的每次输入前缀。它强调：输入 3 个原章节细纲，改编并重排成 2 个新章节大纲。

```text
00C_每次输入前缀_正文生成.md
```

用于正文 GPTS 的每次输入前缀。它强调：每次只写 1 章正文，去掉系统播报、任务面板、世界线修正等科幻感表达。

```text
00_每次输入前缀_独立小世界.md
```

旧文件名兼容版，内容等同于 `00B_每次输入前缀_改编重排.md`，新流程优先使用 `00A/00B/00C`。

```text
01_GPTS_拆原剧情细纲.md
```

放进“拆大纲 GPTS”的系统提示词。输入通常是 3 个连续原文章节，输出 3 个原章节细纲。每个原章节必须 8-10 条节点，不能合并。

```text
02_GPTS_改编新世界细纲.md
```

放进“改编 GPTS”的系统提示词。输入一个拆大纲批次文件，输出 1-2 个新章节大纲。常规情况是 3 个原章节细纲压成 2 个新章节大纲。

输出必须保留稳定标题：

```text
## 新章节大纲

### 新第1章：章节标题
1. ...

### 新第2章：章节标题
1. ...

## 下批承接摘要
...
```

本地拆分脚本依赖 `### 新第1章`、`### 新第2章` 这样的标题识别章节。

```text
03_GPTS_正文生成.md
```

放进“正文 GPTS”的系统提示词。输入单章新大纲，每次只输出 1 章正文。

## 推荐运行流程

### 1. 启动 Chrome

```powershell
cd C:\Users\Administrator\Desktop\xxsbooks
npm run chrome
```

### 2. 预览队列

```powershell
npm run pipeline:dry
```

### 3. 完整运行

```powershell
npm run pipeline
```

完整 pipeline 实际执行：

```powershell
npm run dagang
npm run adapt
npm run split-adapt
npm run zhengwen
```

### 4. 单独运行

拆大纲：

```powershell
npm run dagang
```

改编重排：

```powershell
npm run adapt
```

拆分改编输出：

```powershell
npm run split-adapt
```

正文生成：

```powershell
npm run zhengwen
```

## 质量控制思路

旧流程的问题：

- 单章对单章，章节节奏太像原文。
- 改编大纲仍可能残留系统、任务、奖励、剧情偏移等跨世界词。
- 正文容易像大纲翻译，AI 味重，甚至出现冷冰冰的科幻感。

新版流程的解决方向：

- 拆大纲阶段：一次输入 3 章，但逐章拆出 3 个原细纲，保留原文动作、视线、停顿、试探、误会、关系变化。
- 改编阶段：把 3 个原章细纲重排成 2 个新章细纲，改章节断点、密度、爽点位置和钩子。
- 正文阶段：每次只写 1 章，把大纲揉进场景、动作、对话和人物反应里，避免任务播报和设定讲解。

## 已做的项目优化

- `config-dagang.json` 已改为 `chaptersPerPrompt: 3`。
- `config-adapt.json` 已改为 3 原章细纲 -> 2 新章细纲的 prompt。
- `config-zhengwen.json` 已改为读取 `output/02_adapt_chapters`。
- `config-dagang.json`、`config-adapt.json`、`config-zhengwen.json` 已通过 `promptPrefixFile` 分别引用拆、改编、正文的每次输入前缀，队列实际发送时会自动带上。
- 新增 `scripts/split-adapt-chapters.mjs`，负责把改编输出拆成单章大纲文件。
- `package.json` 的 `pipeline` 已改为：`dagang -> adapt -> split-adapt -> zhengwen`。
- 队列等待逻辑加入了 `maxStableGeneratingMs`，减少正文阶段因页面状态卡住而长时间等待。
- 失败重试时，如果找不到可编辑的上一条用户消息，会自动退回普通发送。

## 下一步测试建议

用户把 3 个 GPTS 系统提示词复制到对应 GPTS 后，先跑一个小批次：

```powershell
npm run dagang -- --limit 1
npm run adapt -- --limit 1
npm run split-adapt
npm run zhengwen -- --limit 2
```

然后检查：

- 拆大纲是否真的 3 个原章节都拆出来；
- 改编是否真的输出 2 个新章节大纲；
- 本地是否拆成 2 个单章文件；
- 正文是否每次只生成 1 章；
- 正文是否比旧版更自然、更少 AI 味和科幻感。
