# XXSBooks 项目说明书

> 本文件合并“项目功能逻辑说明”和“运行/变更日志”。最后更新：2026-07-19。

## 一、项目目标

项目按六条固定生产线管理网络小说。每月在 1、11、21 日开始三个周期，每个周期处理六本书；单本先生产 60 章，再按每天 4 章的排期发布到与该书绑定的番茄账号。到评估点读取成绩：表现好的书继续写，表现不好的书停用后再从素材库选择新书替换，历史书、账号和记录不混用。

## 二、完整功能逻辑

```text
D:\素材库（只读来源）
  → 显式导入一本书并记录来源
  → 为书籍创建独立 config/books/*.json
  → chai：拆改编，输出“拆分”
  → xie：按原章名写作，输出“正文”
  → 质量、章序、账号和排期预检
  → 封面/简介/标签资料预览与送审
  → 番茄正文按绑定账号、作品 ID 顺序发布
  → 记录远端章节 ID、审核状态、排期和证据
  → 到评估日继续或替换
```

关键约束：

- 每本书只能使用自己的 GPTS 对话、正文目录、番茄账号、作品 ID 和排期。
- 番茄正文来源固定为 `书籍/<书名>/正文`，不会从“原文”或“拆分”上传。
- 章节按编号严格连续，远端标题与本地标题不一致时立即停止。
- 番茄账号登录态仅保存在本机独立 Chrome Profile；Cookie、Profile 和素材库不进入 Git。
- 新书默认参加当前可用征文，以 `contestParticipation: true` 明确记录。
- `aiUsed` 必须按正文的真实创作过程明确配置。GPTS 生成或实质改写的正文应声明为使用 AI，项目不会为规避平台披露而伪装成未使用 AI。

## 三、目录与数据职责

| 路径 | 职责 | 是否提交 Git |
|---|---|---|
| `config/books/*.json` | 每本书的生产、账号、作品和排期绑定 | 是 |
| `config/fanqie-marketing.json` | 封面、简介、主角和标签的中央真源 | 是 |
| `config/local/fanqie-accounts.json` | 本机 Chrome Profile 与端口 | 否 |
| `书籍/<书名>/原文` | 导入材料 | 否 |
| `书籍/<书名>/拆分` | chai 输出、xie 输入 | 否 |
| `书籍/<书名>/正文` | 最终章节及番茄资料副本 | 否 |
| `书籍/.state` | 队列、发布、锁、证据和运行日志 | 否 |
| `research` | 榜单与后台核验截图 | 否 |

每本正文目录会保留：

- `番茄封面.png`：平台版 600×800 封面。
- `番茄书籍信息.md`：后台书名、作者、账号、作品 ID、主角、标签和简介。

这两个文件由中央配置生成，不手工分叉维护。

## 四、常用操作

### 4.1 查看生产队列

```powershell
node control.mjs status --json
```

启动、恢复、停止或修复前都先查看此状态。`force`、重置状态和 `reconcile --apply` 只能在明确授权后使用。

### 4.2 查看番茄状态

```powershell
npm run fanqie:local-status
npm run fanqie:status -- --book "书名"
```

多本书启用时必须指定 `--book`。远端状态会核对登录账号、作品、60 章标题、审核状态、排期和本地持久记录。

如果书籍已经改绑到可用账号，但月度投放线仍保留旧账号，先预览再应用受保护的对齐命令：

```powershell
node control.mjs campaign align-accounts
node control.mjs campaign align-accounts --apply
```

### 4.3 同步本地封面与书籍信息

```powershell
npm run fanqie:marketing-sync
npm run fanqie:marketing-sync -- --book "书名"
```

只会写 `正文/番茄封面.png` 和 `正文/番茄书籍信息.md`，不触碰章节。

### 4.4 修改番茄封面、简介和标签

```powershell
# 先预览，不连接后台写入
npm run fanqie:marketing -- --book "书名"

# 核对账号、作品 ID、书名、封面和标签后再应用
npm run fanqie:marketing -- --book "书名" --apply
```

自动化提交前会回读全部表单字段。提交后的判定：

- `visible`：简介与全部标签已经在作品页显示。
- `pending_review`：出现“修改成功”或平台锁定“修改”按钮，表示资料已送审；不得重复提交。
- `uncertain`：既没有成功提示，也没有审核锁定，而且字段未显示；保存截图/HTML 后停止。

平台可能先显示新标签、稍后显示新简介和封面。核验不能只靠立即刷新一次，也不能因旧简介仍可见就再次批量覆盖。

### 4.5 发布正文

```powershell
# 预览计划
npm run fanqie:upload -- --book "书名"

# 明确确认后发布
npm run fanqie:upload -- --book "书名" --apply
```

发布流程固定选择基础内容检测，并按每本书配置中的 `aiUsed` 选择真实 AI 声明。账号级锁保证同一账号不会并发操作；任一章状态不确定时立即停止后续章节。

## 五、弹窗与审核处理规则

浏览器会自动处理编辑器新手引导，以及文字严格匹配以下白名单的非关键弹窗：`我知道了`、`知道了`、`好的`、`下次再说`、`以后再说`、`暂不`、`跳过`、`关闭`、`完成`。

以下按钮不做通用自动点击：`确认`、`提交`、`确认发布`、`完结`、`删除`、`取消`。原因是后台会预加载隐藏的“确认完结”等模态框，仅按 `.arco-modal` 查找并点击通用按钮可能触发破坏性操作。关键弹窗必须由对应业务步骤和精确页面上下文处理。

封面上传必须满足：PNG/JPG/JPEG、600×800、文件不超过 5MB、书名和作者清晰。选择文件后要等待裁切对话框的“确定”按钮变为可用，再提交整张作品信息表单。

## 六、故障恢复

- “修改”按钮变灰：通常是封面或资料正在审核，记录为 `pending_review`，等待审核完成。
- 标签已变、简介仍旧：先检查成功提示和按钮锁定；属于审核延迟时不要重提。
- 登录失效：只在该书绑定的独立 Chrome Profile 中重新登录，不结束其他 Chrome 进程。
- 页面结构变化：保存失败证据并停止；先更新页面契约和测试，不使用坐标盲点。
- 章节提交状态不确定：运行 `npm run fanqie:reconcile -- --book "书名"` 预览，人工核对后才可加 `--apply`。
- 发现锁文件：检查其中 PID 和进程是否仍存活；不得直接删除。
